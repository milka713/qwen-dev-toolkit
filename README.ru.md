# qwen-dev-toolkit

Набор практик для [qwen-code](https://github.com/QwenLM/qwen-code) на **локальных
моделях с маленьким контекстом** (например, llama.cpp с окном ~90–100k). Не даёт большим
сборкам ломаться после переполнения и лоссового компакта: делает модель
**архитектором, который делегирует**, а состояние — переживающим компакт.

> 🇺🇸 [English](README.md) · 🇷🇺 Русский

## Установка / обновление

**Установка и обновление — это одна и та же команда**: запусти ещё раз, и оно обновится на
месте. Копирует только собственные файлы набора в `~/.qwen`; твои другие скиллы, настройки,
API-ключи и память не трогает. Кроссплатформенно: **macOS · Linux · Windows** (один Node-
установщик — сам ставит bash-бэкенды на macOS/Linux и Node-бэкенды на Windows).

### 🤖 Если ты ИИ-агент

Пользователь дал тебе этот репозиторий и просит установить/обновить скиллы? Следуй
**[`INSTALL_FOR_AI.md`](INSTALL_FOR_AI.md)** — пошаговая инструкция специально для тебя,
включая **проверку**, что всё встало. Кратко (и для установки, и для обновления):

```
git clone --depth 1 https://github.com/milka713/qwen-dev-toolkit
node qwen-dev-toolkit/install.js
```

Уже внутри клона? просто `node install.js`. Обновляешь существующий клон? `git pull`, затем
повтори запуск. Уже работаешь внутри qwen-code? вызови **`/toolkit-update`** — сделает всё это за тебя.

### 🧑 Если ты человек

Требуется: **Node.js** + **qwen-code** (и **git** для git-фич). Установщик проверяет их и
пишет, чего не хватает.

```bash
./install.sh      # macOS / Linux
install.cmd       # Windows        (или: node install.js — где угодно)
```

Затем **перезапусти qwen-code**. Чтобы обновиться позже — повтори ту же команду (или
`/toolkit-update` изнутри qwen-code). Удалить: `./uninstall.sh` / `uninstall.cmd`.

**Проверка** (после рестарта): `/skills` показывает `brainstorm, plan, implement, checkpoint,
gitflow, audit, review, commit, docs, changelog, toolkit-update`; `/agents manage` —
`implementer, scout, debugger`; `/status` отвечает.

## Как работает

- **Делегирование** — главная сессия планирует и делегирует; вся тяжёлая работа (чтение
  файлов, написание кода, тесты) идёт в **одноразовых контекстах субагентов**, поэтому
  основной контекст мал и сборка доходит до конца. Субагентов сколько нужно, по одному на задачу.
- **Durable-состояние** — прогресс в `.qwen/PROGRESS.md`, хук переинжектит его каждую
  сессию (и после компакта), так что план не теряется.
- **Флаги, неуязвимые к компакту** — компакт сжимает только *переписку*; `QWEN.md`
  переподключается как системный контекст на каждый запрос. Поэтому блок в `QWEN.md`
  (например флаг режима разработки) всегда на месте и не требует переобъявления.

## Компоненты

### Команды (детерминированные, вызываются вручную)
Custom-команды — запись в файл идёт через shell-шаг (`!{…}`), поэтому детерминирована.

| Команда | Что делает |
| ------- | ---------- |
| `/dev` · `on` · `off` · `status` · `<цель>` | Переключатель режима разработки. `on` пинит блок в `QWEN.md` → сессия планирует и делегирует субагентам `implementer`; `off` убирает; `<цель>` включает **и** строит. Идемпотентно; остальной `QWEN.md` сохраняется. |
| `/cover` · `<N>` · `off` · `status` | Режим test-first / покрытия. Сборка **test-first** (red→green→refactor); покрытие **измеряется** и должно достигать цели — `/cover 95` ставит 95%, просто `/cover` = **80%** по умолчанию. Проверенный результат, не «тыква». |
| `/maxagents <N>` · `off` · `status` | **Жёсткий** лимит, сколько субагентов работает **одновременно**, для слабого локального железа (`/maxagents 1` = строго последовательно). Энфорсится детерминированно хуком `PreToolUse`, который **блокирует** лишние запуски `agent` — не просто инструкция. По умолчанию — без лимита. |
| `/pin <что угодно>` · `list` · `remove <текст>` · `clear` | Запомнить любую инфу (IP/порт, команду деплоя, решение, URL, сниппет) в непробиваемый компактом `FACTS.md` — автозагрузка через `@FACTS.md`, в `.gitignore` (не утечёт). Проектный. |
| `/status` | Read-only снимок: режим dev, цель покрытия, лимит субагентов, число запиненных фактов, текущая цель и следующая задача. |
| `/main-push` · `off` · `status` | Разрешить push/merge в защищённую ветку (`main`/`master`) на **15-минутное окно релиза** (покрывает и merge, и push). Хук `git-branch-guard` по умолчанию блокирует операции с main; `/main-push` — вентиль, доступный только пользователю. |
| `/versioning` · `on` · `off` · `status` · `<свой способ>` | Переключатель именования версий (**по умолчанию выключен**). On → называть версии по semver-значимости (PATCH `1.4.7` / MINOR `1.5.0` / MAJOR `2.0.0`) и говорить, что именно бампнул. Передай текст — закрепит свой способ. Пин **глобально**. |
| `/bro` · `свобода` · `ламар` · `off` · `status` | Общаться по-братски, в одной из **двух персон**: `свобода` — свободовец из S.T.A.L.K.E.R. (всегда зовёт тебя «мэн»), `ламар` — уличный кореш в духе Ламара из GTA V («homie/foo/dog») — по-свойски, со сленгом, блюнтово, но реально по делу. По умолчанию выключен; пин **глобально** до `/bro off`. |

### Скиллы (вызываются и моделью, и вручную)

| Скилл | Что делает |
| ----- | ---------- |
| `/brainstorm` | Проясняет и прожимает требования **до** кода (объём, критерии готовности, edge-кейсы, ограничения), чтобы не потратить маленький контекст на «не то». Выдаёт согласованный спек и передаёт в `/plan`. |
| `/plan` | Превращает размытый/большой запрос в упорядоченный по зависимостям список задач в `.qwen/PROGRESS.md`. Разведка через read-only `scout`. Выдаёт план, не код. |
| `/implement` | Оркестратор: фиксирует цель, декомпозирует, гоняет **каждую задачу в свежем субагенте `implementer`**, затем проверяет end-to-end канонической командой. Для любой многошаговой сборки. |
| `/checkpoint` | Собирает важное состояние в `.qwen/PROGRESS.md`, чтобы пережить лоссовое авто-сжатие. `/checkpoint restore` перезагружает. |
| `/gitflow` | Дисциплина веток и деплоя, применяется проактивно при любом commit/push/merge/deploy: **новая работа → ветка `dev` по умолчанию; `main`/`master` — только по явному разрешению**. Подкреплено на уровне движка хуком `git-branch-guard` + `/main-push`. |
| `/audit` | Аудит безопасности (архитектура + код): хардкод-секреты, authz, инъекции, SSRF, слабая крипта, рискованные зависимости — находки по серьёзности, чинит безопасные. |
| `/review` | Проход по текущему диффу на корректность и качество — баги, краевые случаи, несоответствия контракту, мёртвый/переусложнённый код (отдельно от `/audit`, который про безопасность). Находки по серьёзности, чинит безопасные, перегоняет тесты. |
| `/commit` | Осознанно стейджит и пишет чистое сообщение в стиле Conventional Commits **из реального диффа**; уважает `gitflow` (ложится в `dev`, не в `main` без релиза). |
| `/docs` | Держит доки в синхроне с кодом после изменений — `README.md` **и** `README.ru.md` (двуязычный паритет), примеры, help-текст — точность важнее полноты. |
| `/changelog` | Собирает человеческую запись в `CHANGELOG.md` из git-лога с последнего тега, сгруппированно в стиле Keep a Changelog (Added/Changed/Fixed/…). |
| `/toolkit-update` | **Ставит или обновляет сам этот набор** из GitHub одной командой (скачать + запустить кроссплатформенный установщик + проверить). Установка и обновление — одна команда. |

### Субагенты (изолированный контекст)

| Субагент | Что делает |
| -------- | ---------- |
| `implementer` | Доводит **одну** задачу до проверенного состояния: читает реальные файлы, реализует полностью (без заглушек), проверяет **канонической** командой из корня (чинит упаковку, если проходит лишь через трюк с путём). Возвращает краткое резюме. |
| `scout` | Read-only исследователь — отдаёт компактный дайджест (ключевые файлы, связи, конвенции, реальные команды) вместо чтения файлов пачками в основной контекст. |
| `debugger` | Дебаггер по первопричине — воспроизводит падающий тест/ошибку в своём контексте, находит *настоящую* причину (не симптом), делает минимальный фикс, проверяет репро + сьют, возвращает диагноз. |

### Хуки (`~/.qwen/settings.json`)

| Хук | Что делает |
| --- | ---------- |
| `SessionStart` → `session-start-restore.js` | Переинжектит `.qwen/PROGRESS.md` в начале сессии / после компакта — модель восстанавливает цель и следующие шаги. |
| `PreCompact` → `pre-compact-steer.js` | Направляет встроенное сжатие сохранять цель, решения, список файлов и разбивку сделано/осталось. |
| `PreToolUse` → `secret-guard.js` | **Блокирует** любую запись/правку/команду с хардкод-секретом (приватные ключи, токены AWS/OpenAI/GitHub/Slack/HF, …) или коммит секрет-файла (`.env`, `id_rsa`, `*.pem`). Env-переменные и плейсхолдеры проходят. |
| `PreToolUse` → `git-branch-guard.js` | **Блокирует** любой `git push`/`merge`/`rebase`, затрагивающий `main`/`master` (явная цель, или когда ты на ней, или one-liner «switch + merge»). Пуши в `dev`/feature-ветки и read-only git проходят. Разблокируется на одну операцию через `/main-push`. |
| `UserPromptSubmit` → `skill-reminder.js` | Маленькие локальные модели недо-вызывают скиллы; хук инжектит короткую точечную подсказку (напр. «похоже на безопасность → `/audit`) только при явном совпадении, чтобы нужный скилл реально срабатывал. На тривиальных промптах молчит. |
| `PreToolUse`/`PostToolUse`/`SessionStart` → `agent-limit.js` | Детерминированно энфорсит `/maxagents`: считает запущенных субагентов и **блокирует** запуски `agent` сверх лимита (потокобезопасно через лок), декрементит при завершении, сбрасывает каждую сессию. Лимит не задан → ничего не делает. |

Плюс лаконичный `~/.qwen/QWEN.md` (режимы + дисциплина памяти) и нативная авто-память.

## Области видимости

`QWEN.md` — контекст-файл, переподключаемый на каждый запрос (не сжимается); грузится из
`~/.qwen/QWEN.md` (**глобально**) и `<проект>/QWEN.md` вверх по дереву (**проектно**).
Сессионного `QWEN.md` нет.

| Состояние | Где | Область |
| --------- | --- | ------- |
| Скиллы, субагенты, команды, хуки, гайд | `~/.qwen/…` | **Глобально** |
| Память `/pin` | `<проект>/FACTS.md` (в `.gitignore`) | **Проектно** |
| Флаги `/dev` и `/cover` | блок в `<проект>/QWEN.md` | **Проектно** (липко до `off`) |
| Состояние задач | `<проект>/.qwen/PROGRESS.md` | **Проектно** |

## Использование

```text
/brainstorm API-сокращалка ссылок    # сперва зафиксировать объём и критерии готовности
/dev                                 # режим разработки (архитектор + делегирование)
/cover 80                            # test-first, требовать ≥80% измеренного покрытия
/maxagents 2                         # ограничить параллельных субагентов (слабое железо)
/pin модель-сервер 10.0.0.5:8080     # запомнить инфра-факты (компакт-пруф, в .gitignore)
/plan добавь JWT-авторизацию в API   # дизайн → .qwen/PROGRESS.md
/implement                           # сборка через субагентов
/audit                               # аудит безопасности
/checkpoint [restore]                # сохранить / перезагрузить состояние
/status                              # что включено, текущая цель
/dev off                             # обратно в режим одного агента
```

Одной строкой: `/dev собери Python CLI трекер расходов с SQLite и pytest`.

### Типичная сессия от начала до конца

В основном ты говоришь с ним обычным языком — скиллы и защиты срабатывают сами.

```text
> /brainstorm сервис-сокращалка ссылок на Python
    ← согласует объём, критерии успеха и краевые случаи, записывает их

> /dev собери
    ← становится архитектором: планирует модули, делегирует каждый свежему
      implementer-субагенту, отмечая задачи в .qwen/PROGRESS.md по мере готовности.
      Переполнение контекста + компакт посреди сборки? SessionStart перезагрузит
      PROGRESS.md, и работа продолжится с первой невыполненной задачи — ничего не теряется.

> запушь готовое
    ← включается gitflow: создаёт `dev` если нет, коммитит, пушит в origin/dev.
      main не трогается — хук git-branch-guard блокирует это по умолчанию.

> выкати в main
    ← попросит сначала запустить /main-push (main защищён)
> /main-push                   ← открываешь 15-минутное окно релиза
> выкати в main             ← теперь мержит dev → main и пушит
```

Смысл: тебе не нужно помнить правила веток, повторять план после компакта или
следить, какой субагент что делает — это берёт на себя тулкит.

## Надёжность на маленьком / медленном / общем локальном сервере

Две настройки в `~/.qwen/settings.json` сильно влияют на локальную модель и легко
задаются неверно. Для **кастомного OpenAI-провайдера** клади их в `generationConfig`
*записи провайдера* (qwen игнорирует `model.generationConfig` для таких провайдеров):

```json
{
  "modelProviders": {
    "openai": [{
      "id": "...", "name": "...", "baseUrl": "http://HOST:PORT/v1", "envKey": "OPENAI_API_KEY",
      "generationConfig": {
        "contextWindowSize": 120000,
        "timeout": 1800000
      }
    }]
  }
}
```

- **`contextWindowSize`** — qwen сжимает контекст *до* переполнения, но порог считает из
  окна модели. Для кастомного провайдера он его не знает и берёт дефолт, который может быть
  **больше** реального окна → сжимает **слишком поздно** и переполняется. Ставь чуть ниже
  `-c` сервера (напр. `120000` для llama.cpp `-c 125000`). Дальше хук `PreCompact` сохраняет
  цель/решения при сжатии, а `SessionStart` перезагружает `.qwen/PROGRESS.md`.
- **`timeout`** (мс) — **per-request** таймаут. Дефолт (~6 мин) обрывает *один* вызов модели,
  когда сервер медленный или занят другими задачами, убивая сборку на середине (это отдельно
  от общего бюджета прогона). Подними щедро — `1800000` (30 мин) — чтобы долгие генерации под
  нагрузкой завершались, а не падали с `Request timeout`.

### Защита от зацикливания

Маленькие локальные модели иногда застревают, повторяя один и тот же tool-call или вывод.
У qwen-code есть детектор циклов, но он **по умолчанию выключен** (чтобы не было ложных
срабатываний). Для локальной модели его стоит включить, плюс конечный бэкстоп по
tool-call'ам — это настройки верхнего уровня `model`:

```json
{ "model": { "skipLoopDetection": false, "maxToolCalls": 5000 } }
```

- **`skipLoopDetection: false`** включает детектор циклов (повторяющиеся одинаковые
  tool-call'ы или повторяющийся контент в стриме). В интерактиве спрашивает, продолжать ли,
  когда поймал цикл; в headless просто останавливает залипший цикл, не сжигая бюджет.
- **`maxToolCalls: 5000`** — жёсткий бэкстоп: убежавший цикл оборвётся (exit 55) на 5000
  tool-call'ах, а обычная сборка туда и близко не подойдёт.
- Для авто-прогонов добавляй `--max-wall-time 1800` как общий лимит времени. На ходу
  `Esc` (или `Ctrl+C`) прерывает сразу.

Замечание: забытый `/maxagents 1` в проекте сам может спровоцировать цикл — когда модель
пытается запустить несколько субагентов, каждый лишний отклоняется, и маленькая модель
может долбить один и тот же запуск. Сбрось через `/maxagents off`, если не ограничиваешь
специально.

### Auto-режим с очередью / общей моделью

`tools.approvalMode: "auto"` проверяет рискованные действия (shell, запуск субагентов,
запись вне проекта) маленьким LLM-классификатором — безопасное пропускает, рискованное
блокирует/спрашивает. У классификатора **короткие таймауты по умолчанию** (**stage 1 ≈ 10 с,
stage 2 ≈ 30 с**), рассчитанные на быструю выделенную модель. Если твоя локальная модель
стоит за **очередью доступа** (занята другим пользователем), qwen-code не отличает «в
очереди» от «медленно»: вызов классификатора ждёт своей очереди, вылезает за 10 с и
**падает closed** (`Classifier stage 1 unavailable`) — блокируя действие, хотя всё в порядке.

Подними таймауты классификатора под очередь. «Отключить в ноль» нельзя (значения ниже 1000 мс
откатываются к дефолту), поэтому ставь большое значение — оно фактически выключает таймаут,
но реально зависший вызов всё равно оборвёт:

```json
{ "permissions": { "autoMode": { "classifier": {
  "timeouts": { "stage1Ms": 1200000, "stage2Ms": 1200000 }
} } } }
```

(по 20 минут выше.) Два альтернативных пути: указать **fast-модель** (`/model --fast`) на
маленькую быструю модель, чтобы классификатор отвечал за ~300 мс и не ждал основную; либо
вообще без классификатора — `yolo` + чёрный список `permissions.deny` (`deny` приоритетнее
всего даже в yolo).

## Требования

- **qwen-code** (проверено на **0.19.x**) + **Node.js**; **git** для git-фич. Любой провайдер;
  задуман для локальных моделей с маленьким контекстом. Работает на macOS, Linux и Windows.

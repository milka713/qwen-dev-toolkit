# qwen-dev-toolkit

> ### 💬 Если ты ИИ-агент — начни отсюда
>
> Пользователь дал тебе этот репозиторий и просит установить или обновить скиллы? Следуй
> **[`INSTALL_FOR_AI.md`](INSTALL_FOR_AI.md)** — пошаговая инструкция специально для тебя,
> с проверкой результата. Там же — как отключить **stream-idle таймаут** qwen-code
> (обрывает запросы после 240 с тишины стрима — яд для медленных локальных серверов с
> очередью) на macOS, Linux и Windows. Кратко:
>
> ```
> git clone --depth 1 https://github.com/milka713/qwen-dev-toolkit
> node qwen-dev-toolkit/install.js
> ```
>
> Уже внутри клона? просто `node install.js`. Обновляешь? `git pull`, затем повтори запуск.
> Уже работаешь внутри qwen-code? вызови **`/toolkit-update`** — сделает всё это за тебя.

Набор скиллов для [qwen-code](https://github.com/QwenLM/qwen-code) на **локальных
моделях с маленьким контекстом** (например, llama.cpp с окном ~90–100k). Его ядро не даёт
большим сборкам ломаться после переполнения и лоссового компакта: делает модель
**архитектором, который делегирует**, а состояние — переживающим компакт. А сверху — целый
набор скиллов и команд для удобной разработки: дисциплина git (`/gitflow`, `/commit`,
`/main-push`), ревью кода `/review` и аудит безопасности `/audit`, `/docs` и `/changelog`,
планирование (`/brainstorm`, `/plan`), режим покрытия (`/cover`), дебаггер по первопричине
`debugger` и другое — всё перечислено ниже.

## Как работает

- **Делегирование** — главная сессия планирует и делегирует; вся тяжёлая работа (чтение
  файлов, написание кода, тесты) идёт в **одноразовых контекстах субагентов**, поэтому
  основной контекст мал и сборка доходит до конца. Субагентов сколько нужно, по одному на задачу.
- **Durable-состояние** — прогресс в `.qwen/PROGRESS.md`, хук переинжектит его каждую
  сессию (и после компакта), так что план не теряется.
- **Флаги, неуязвимые к компакту** — компакт сжимает только *переписку*; `QWEN.md`
  переподключается как системный контекст на каждый запрос. Поэтому блок в `QWEN.md`
  (например флаг режима разработки) всегда на месте и не требует переобъявления.

## Компоненты

### Команды (детерминированные, вызываются вручную)
Custom-команды — запись в файл идёт через shell-шаг (`!{…}`), поэтому детерминирована.

Каждая хранит состояние в помеченном блоке в `QWEN.md` (или в `FACTS.md`), который
переподключается в контекст на каждый запрос и потому переживает компакт. Аргументы указаны
после имени.

**`/dev` · `on` · `off` · `status` · `<цель>`** — Переключатель режима разработки: делает
сессию **архитектором, который делегирует**. Когда включён, модель планирует работу и отдаёт
каждую задачу свежему субагенту `implementer`, а не пишет код в своём контексте — именно это
позволяет большой сборке дойти до конца, а не переполниться и сломаться после компакта. `on`
пинит блок в `QWEN.md` проекта; `off` убирает; `status` показывает; `/dev <цель>` включает
**и** сразу начинает строить в том же ходу. Идемпотентно — остальной `QWEN.md` не трогается.
· _Пример:_ `/dev собери REST API для задач с SQLite и тестами`

**`/cover` · `<N>` · `off` · `status`** — Режим test-first / покрытия. Пока включён, ничто не
«готово» без проходящих тестов: модель работает **red → green → refactor** и обязана
**измерить** покрытие реальным инструментом проекта (`pytest --cov`, `jest --coverage`, `go
test -cover`, …) и достичь цели. `/cover 95` ставит 95 %, просто `/cover` = 80 %, `/cover off`
снимает. Защита от пустого непроверенного вывода («тыквы»).
· _Пример:_ `/cover 90`

**`/maxagents <N>` · `off` · `status`** — Жёсткий лимит, сколько субагентов работает
**одновременно**, для слабого или загруженного железа. Не просто инструкция: хук `PreToolUse`
считает запущенных и **блокирует** запуск сверх N (`/maxagents 1` = строго последовательно).
По умолчанию лимита нет; забытый сбрось через `/maxagents off` (оставшаяся `1` может вызвать
циклы повторов).
· _Пример:_ `/maxagents 2`

**`/pin <что угодно>` · `list` · `remove <текст>` · `clear`** — Запомнить любой факт, который
нужен всегда под рукой — хост/порт, команду деплоя, решение, URL, сниппет. Ложится в проектный
`FACTS.md`, который автозагружается каждую сессию через include `@FACTS.md` (переживает
компакт) и лежит в `.gitignore`, чтобы не утёк в репозиторий. `list` показывает, `remove
<текст>` убирает совпадающие строки, `clear` чистит всё.
· _Пример:_ `/pin деплой = ssh -p 12578 mark@host && ./deploy-dev.sh`

**`/status`** — Read-only снимок состояния тулкита в проекте: режим dev вкл/выкл, цель
покрытия, лимит субагентов, сколько фактов запинено, и — из `.qwen/PROGRESS.md` — текущая цель
и следующая невыполненная задача. Ничего не меняет.

**`/main-push` · `off` · `status`** — Вентиль релиза в защищённую ветку, доступный только
пользователю. По умолчанию хук `git-branch-guard` блокирует любой push/merge в `main`/`master`;
запуск `/main-push` открывает **15-минутное окно**, разрешающее весь релиз (и merge, и push).
`off` отзывает раньше, `status` проверяет. Так как слэш-команду можешь запустить только ты,
«да, реально катим в main» невозможно подделать модели.

**`/versioning` · `on` · `off` · `status` · `<свой способ>`** — Политика именования версий.
Semver **включён по умолчанию** (прописан в глобальном `QWEN.md`): модель называет версии по
значимости — **PATCH** для мелких фиксов (`1.4.7`), **MINOR** для заметных фич (`1.5.0`),
**MAJOR** для ломающих изменений (`2.0.0`) — и говорит, что именно бампнула. Команда задаёт
**переопределение на проект** в `QWEN.md` проекта: передай текст для своей схемы или `off`,
чтобы отключить semver в этом проекте.
· _Примеры:_ `/versioning` · `/versioning CalVer вида 2026.07`

**`/bro` · `свобода` · `ламар` · `off` · `status`** — Общаться по-братски вместо формального
ассистента, в одной из двух персон: `свобода` — свободовец из S.T.A.L.K.E.R., всегда зовёт
тебя «мэн»; `ламар` — уличный кореш в духе Ламара из GTA V («homie/foo/dog»). По-свойски, со
сленгом и блюнтово, но реально точно и по делу — вайб это обёртка, а не повод сачкануть. По
умолчанию выключен; пин **на проект** до `/bro off`.

### Скиллы (вызываются и моделью, и вручную)

В отличие от команд, эти скиллы модель может вызвать **сама**, когда они уместны (или ты
запускаешь `/имя`). Аргументы в `<…>` необязательны.

**`/brainstorm`** — Фиксирует *что* строить до кода. Проясняет и прожимает требования — объём,
критерии готовности, краевые случаи, ограничения, что явно вне рамок — чтобы не потратить
маленький контекст на «не то». Выдаёт согласованный спек, сохраняет его в `.qwen/PROGRESS.md`
(переживает компакт — история чата нет) и передаёт в `/plan`.

**`/plan`** — Превращает размытый или большой запрос в конкретный, **упорядоченный по
зависимостям список задач** в `.qwen/PROGRESS.md`, сперва изучив незнакомый код через
read-only субагента `scout`. Выдаёт план, не код — durable-точку, с которой сборка продолжится
после любого рестарта или компакта.

**`/implement`** — Оркестратор любой многошаговой сборки. Фиксирует цель, декомпозирует на
right-sized задачи (≈ один модуль + его тесты), гоняет **каждую в свежем субагенте
`implementer`**, отмечает в `.qwen/PROGRESS.md`, и завершает end-to-end проверкой канонической
командой проекта — реально **запуская** каждую названную точку входа, а не просто импортируя.
Делегирование вместо инлайн-кода и позволяет большим проектам дойти до конца на маленьком
контексте.

**`/checkpoint` · `restore`** — Собирает важное состояние (цель, решения, карту файлов,
сделано/осталось) в `.qwen/PROGRESS.md`, чтобы пережить лоссовое авто-сжатие; `/checkpoint
restore` перезагружает его в контекст после компакта или в новой сессии.

**`/gitflow`** — Дисциплина веток и деплоя, применяется проактивно при любом
commit/push/merge/deploy: **новая работа → `dev` по умолчанию; `main`/`master` — только по
твоему явному разрешению**, со здравым порядком деплоя (dev → тест → подтверждение → main →
prod). Подкреплено на уровне движка хуком `git-branch-guard` и разблокируется через
`/main-push`.

**`/audit`** — **Аудит безопасности** архитектуры и кода: хардкод-секреты, сломанная authz,
инъекции, SSRF, слабая крипта, рискованные зависимости. Находки по серьёзности, чинит явно
безопасные. Запускай перед выкаткой всего, что касается авторизации, сети, файлов, секретов
или БД.

**`/review` · `<путь>`** — Проход на **корректность и качество** по текущему диффу (в отличие
от безопасности в `/audit`): реальные баги, недообработанные краевые случаи, несоответствия
контракту/спеке, мёртвый или переусложнённый код. Находки по серьёзности, чинит безопасные
однозначные и перегоняет тесты, подтверждая зелёные.

**`/commit` · `<подсказка>`** — Осознанно стейджит нужные файлы (не слепой `git add -A`) и
пишет чистое сообщение в стиле **Conventional Commits** из реального диффа, а не из догадки.
Уважает `gitflow` — коммитит в `dev`/feature-ветку, не прямо в `main` — не коммитит секреты и
не пушит без твоей просьбы.

**`/docs` · `<что изменилось>`** — Держит документацию в синхроне с кодом после изменений:
`README.md` **и** `README.ru.md` в двуязычном паритете, примеры, которые реально работают,
help-текст и таблицы команд. Точность важнее полноты — сверяет имена/флаги с кодом и не
документирует несуществующее.

**`/changelog` · `<версия>`** — Собирает человекочитаемую запись в `CHANGELOG.md` из git-лога с
последнего тега, сгруппированно в стиле Keep a Changelog (Added / Changed / Fixed / …),
переписывая сабджекты коммитов в понятные строки и предлагая следующий semver. Опирается на
реальные коммиты — без выдуманных пунктов.

**`/release` · `check` · `<версия>`** — Нарезает релиз так, чтобы опубликованный тег / GitHub
Release не отставал от кода. Ловит рассинхрон (последний тег vs файл `VERSION` vs коммиты на
`main`) и, когда бамп готов, создаёт аннотированный git-тег и GitHub Release с заметками из
`CHANGELOG.md`. `/release check` показывает состояние синхронизации, ничего не меняя; и
отказывается релизить устаревший код (коммиты после тега) или из `dev`.

**`/toolkit-update`** — Ставит или обновляет **сам этот набор** из GitHub одной командой:
скачивает свежую версию, запускает кроссплатформенный установщик и проверяет. Установка и
обновление — одна операция; работает из любого места (нужны `git` + `node`).

### Субагенты (изолированный контекст)

| Субагент | Что делает |
| -------- | ---------- |
| `implementer` | Доводит **одну** задачу до проверенного состояния: читает реальные файлы, реализует полностью (без заглушек), проверяет **канонической** командой из корня (чинит упаковку, если проходит лишь через трюк с путём). Возвращает краткое резюме. |
| `scout` | Read-only исследователь — отдаёт компактный дайджест (ключевые файлы, связи, конвенции, реальные команды) вместо чтения файлов пачками в основной контекст. |
| `debugger` | Дебаггер по первопричине — воспроизводит падающий тест/ошибку в своём контексте, находит *настоящую* причину (не симптом), делает минимальный фикс, проверяет репро + сьют, возвращает диагноз. |
| `tester` | Независимый **black-box** проверяющий — выводит проверки из спеки (не из кода) и прогоняет каждый критерий готовности буквально, из чистого корня репо; ловит то, что пропускают тесты, написанные самим implementer (обещанный экспорт, которого нет; CLI, который не запускается). Отчёт PASS/FAIL по каждому критерию, ничего не чинит. |
| `researcher` | Read-only исследователь библиотек/API — фиксирует версию, которую реально использует проект, сверяется с официальной документацией и локально установленным пакетом, возвращает компактный дайджест (настоящие сигнатуры, один рабочий пример, подводные камни) — чтобы никто не писал код по полузабытому API. |
| `verifier` | Состязательный проверяющий **одного** утверждения (подозрение на баг, уязвимость или «требование выполнено») — сначала пытается его *опровергнуть* (ищет валидатор/вызывающий код/тест, который случай обрабатывает), возвращает CONFIRMED / REFUTED / PLAUSIBLE с доказательствами file:line. Валидирует кандидатов из `/review` и `/audit`. |

### Хуки (`~/.qwen/settings.json`)

| Хук | Что делает |
| --- | ---------- |
| `SessionStart` → `session-start-restore.js` | Переинжектит `.qwen/PROGRESS.md` в начале сессии / после компакта — модель восстанавливает цель и следующие шаги. |
| `PreCompact` → `pre-compact-steer.js` | Направляет встроенное сжатие сохранять цель, решения, список файлов и разбивку сделано/осталось. |
| `PreToolUse` → `secret-guard.js` | **Блокирует** любую запись/правку/команду с хардкод-секретом (приватные ключи, токены AWS/OpenAI/GitHub/Slack/HF, …) или коммит секрет-файла (`.env`, `id_rsa`, `*.pem`). Env-переменные и плейсхолдеры проходят. |
| `PreToolUse` → `git-branch-guard.js` | **Блокирует** любой `git push`/`merge`/`rebase`, затрагивающий `main`/`master` (явная цель, или когда ты на ней, или one-liner «switch + merge»). Пуши в `dev`/feature-ветки и read-only git проходят. Разблокируется на одну операцию через `/main-push`. |
| `PreToolUse` → `release-guard.js` | **Напоминает** (не блокирует), когда пуш продвигает `main`/`master`, а релиз отстаёт от кода — бампнутый `VERSION` без тега, или коммиты после тега без бампа — подсовывая напоминание запустить `/release` (или `/changelog`, затем `/release`). Это детерминированный бэкстоп, из-за которого `/release` сработает, даже если модель про него забыла. Молчит, когда релиз в синхроне. |
| `UserPromptSubmit` → `skill-reminder.js` | Маленькие локальные модели недо-вызывают скиллы; хук инжектит короткую точечную подсказку (напр. «похоже на безопасность → `/audit`) только при явном совпадении, чтобы нужный скилл реально срабатывал. На тривиальных промптах молчит. |
| `PreToolUse`/`PostToolUse`/`SessionStart` → `agent-limit.js` | Детерминированно энфорсит `/maxagents`: считает запущенных субагентов и **блокирует** запуски `agent` сверх лимита (потокобезопасно через лок), декрементит при завершении, сбрасывает каждую сессию. Лимит не задан → ничего не делает. |

Плюс лаконичный `~/.qwen/QWEN.md` (режимы + дисциплина памяти) и нативная авто-память.

## Области видимости

`QWEN.md` — контекст-файл, переподключаемый на каждый запрос (не сжимается); грузится из
`~/.qwen/QWEN.md` (**глобально**) и `<проект>/QWEN.md` вверх по дереву (**проектно**).
Сессионного `QWEN.md` нет.

| Состояние | Где | Область |
| --------- | --- | ------- |
| Скиллы, субагенты, команды, хуки, гайд | `~/.qwen/…` | **Глобально** |
| Память `/pin` | `<проект>/FACTS.md` (в `.gitignore`) | **Проектно** |
| Флаги `/dev`, `/bro`, `/cover`, `/versioning` | блок в `<проект>/QWEN.md` | **Проектно** (липко до `off`) |
| Состояние задач | `<проект>/.qwen/PROGRESS.md` | **Проектно** |

## Использование

```text
/brainstorm API-сокращалка ссылок    # сперва зафиксировать объём и критерии готовности
/dev                                 # режим разработки (архитектор + делегирование)
/cover 80                            # test-first, требовать ≥80% измеренного покрытия
/maxagents 2                         # ограничить параллельных субагентов (слабое железо)
/pin модель-сервер 10.0.0.5:8080     # запомнить инфра-факты (компакт-пруф, в .gitignore)
/plan добавь JWT-авторизацию в API   # дизайн → .qwen/PROGRESS.md
/implement                           # сборка через субагентов
/audit                               # аудит безопасности
/checkpoint [restore]                # сохранить / перезагрузить состояние
/status                              # что включено, текущая цель
/dev off                             # обратно в режим одного агента
```

Одной строкой: `/dev собери Python CLI трекер расходов с SQLite и pytest`.

## Надёжность на маленьком / медленном / общем локальном сервере

Две настройки в `~/.qwen/settings.json` сильно влияют на локальную модель и легко
задаются неверно. Для **кастомного OpenAI-провайдера** клади их в `generationConfig`
*записи провайдера* (qwen игнорирует `model.generationConfig` для таких провайдеров):

```json
{
  "modelProviders": {
    "openai": [{
      "id": "...", "name": "...", "baseUrl": "http://HOST:PORT/v1", "envKey": "OPENAI_API_KEY",
      "generationConfig": {
        "contextWindowSize": 120000,
        "timeout": 1800000
      }
    }]
  }
}
```

- **`contextWindowSize`** — qwen сжимает контекст *до* переполнения, но порог считает из
  окна модели. Для кастомного провайдера он его не знает и берёт дефолт, который может быть
  **больше** реального окна → сжимает **слишком поздно** и переполняется. Ставь чуть ниже
  `-c` сервера (напр. `120000` для llama.cpp `-c 125000`). Дальше хук `PreCompact` сохраняет
  цель/решения при сжатии, а `SessionStart` перезагружает `.qwen/PROGRESS.md`.
- **`timeout`** (мс) — **per-request** таймаут. Дефолт (~6 мин) обрывает *один* вызов модели,
  когда сервер медленный или занят другими задачами, убивая сборку на середине (это отдельно
  от общего бюджета прогона). Подними щедро — `1800000` (30 мин) — чтобы долгие генерации под
  нагрузкой завершались, а не падали с `Request timeout`.

### Stream-idle таймаут (запросы умирают ровно на 240-й секунде тишины)

qwen-code ≥ 0.19.3 обрывает стриминговый запрос после **240 с без нового SSE-чанка**
(120 с в 0.19.3–0.19.7) с ошибкой вида
`[API Error: No stream activity for 240000ms after 1 chunks]`. На общем локальном сервере
такая тишина обычно **легитимна** — запрос стоит в FIFO-очереди сервера или идёт prefill
100k-контекста, — а авторетрай просто встаёт в ту же очередь заново, и здоровые запросы
гибнут один за другим. Опции в settings.json **нет** (per-provider настройка предложена в
[QwenLM/qwen-code#5975](https://github.com/QwenLM/qwen-code/issues/5975)); управляется
только переменной окружения `QWEN_STREAM_IDLE_TIMEOUT_MS` — `0` отключает, любое другое
значение = новое окно в мс (общий `timeout` выше по-прежнему ограничивает весь вызов):

- **macOS (zsh):** `echo 'export QWEN_STREAM_IDLE_TIMEOUT_MS=0' >> ~/.zshrc`, затем новый
  терминал (или `source ~/.zshrc`).
- **Linux / Ubuntu-сервер (bash):** `echo 'export QWEN_STREAM_IDLE_TIMEOUT_MS=0' >> ~/.bashrc`,
  затем перелогин. Для qwen под systemd-юнитом — вместо этого добавь в юнит
  `Environment=QWEN_STREAM_IDLE_TIMEOUT_MS=0`.
- **Windows:** PowerShell:
  `[Environment]::SetEnvironmentVariable('QWEN_STREAM_IDLE_TIMEOUT_MS','0','User')`
  (или cmd: `setx QWEN_STREAM_IDLE_TIMEOUT_MS 0`), затем новый терминал.

### Защита от зацикливания

Маленькие локальные модели иногда застревают, повторяя один и тот же tool-call или вывод.
У qwen-code есть детектор циклов, но он **по умолчанию выключен** (чтобы не было ложных
срабатываний). Для локальной модели его стоит включить, плюс конечный бэкстоп по
tool-call'ам — это настройки верхнего уровня `model`:

```json
{ "model": { "skipLoopDetection": false, "maxToolCalls": 5000 } }
```

- **`skipLoopDetection: false`** включает детектор циклов (повторяющиеся одинаковые
  tool-call'ы или повторяющийся контент в стриме). В интерактиве спрашивает, продолжать ли,
  когда поймал цикл; в headless просто останавливает залипший цикл, не сжигая бюджет.
- **`maxToolCalls: 5000`** — жёсткий бэкстоп: убежавший цикл оборвётся (exit 55) на 5000
  tool-call'ах, а обычная сборка туда и близко не подойдёт.
- Для авто-прогонов добавляй `--max-wall-time 1800` как общий лимит времени. На ходу
  `Esc` (или `Ctrl+C`) прерывает сразу.

Замечание: забытый `/maxagents 1` в проекте сам может спровоцировать цикл — когда модель
пытается запустить несколько субагентов, каждый лишний отклоняется, и маленькая модель
может долбить один и тот же запуск. Сбрось через `/maxagents off`, если не ограничиваешь
специально.

### Auto-режим с очередью / общей моделью

`tools.approvalMode: "auto"` проверяет рискованные действия (shell, запуск субагентов,
запись вне проекта) маленьким LLM-классификатором — безопасное пропускает, рискованное
блокирует/спрашивает. У классификатора **короткие таймауты по умолчанию** (**stage 1 ≈ 10 с,
stage 2 ≈ 30 с**), рассчитанные на быструю выделенную модель. Если твоя локальная модель
стоит за **очередью доступа** (занята другим пользователем), qwen-code не отличает «в
очереди» от «медленно»: вызов классификатора ждёт своей очереди, вылезает за 10 с и
**падает closed** (`Classifier stage 1 unavailable`) — блокируя действие, хотя всё в порядке.

Подними таймауты классификатора под очередь. «Отключить в ноль» нельзя (значения ниже 1000 мс
откатываются к дефолту), поэтому ставь большое значение — оно фактически выключает таймаут,
но реально зависший вызов всё равно оборвёт:

```json
{ "permissions": { "autoMode": { "classifier": {
  "timeouts": { "stage1Ms": 1200000, "stage2Ms": 1200000 }
} } } }
```

(по 20 минут выше.) Два альтернативных пути: указать **fast-модель** (`/model --fast`) на
маленькую быструю модель, чтобы классификатор отвечал за ~300 мс и не ждал основную; либо
вообще без классификатора — `yolo` + чёрный список `permissions.deny` (`deny` приоритетнее
всего даже в yolo).

**Reasoning-модель ломает классификатор ещё одним способом** — и поднятие таймаута тут не
поможет. Stage 1 ограничивает ответ ~32 токенами, но думающая модель тратит их на фазу
`<think>` и возвращает **пустой контент** → парсится как `unavailable` → блок. Классификатор
(или fast-модель под ним) обязан отвечать *без* размышления. На сервере llama.cpp это
серверный дефолт (`--chat-template-kwargs '{"enable_thinking":false}'`, который заодно лишает
размышления и **основной** агент) или **отдельная не-reasoning fast-модель**. Учти: маленькие
специализированные guard-модели (Llama Guard, Qwen3Guard, ShieldGemma) классифицируют
*вредность контента*, а не *опасность команд* — `rm -rf /` они не поймают. Поэтому если хочешь
сохранить размышление основного агента, надёжный путь — **`yolo` + усиленный `permissions.deny`**
(затирание дисков, `dd`, `mkfs`, pipe-в-шелл, чтение ключей, …) вместе с хуками `secret-guard` /
`git-branch-guard`, которые срабатывают в **любом** режиме, включая yolo.

## Требования

- **qwen-code** (проверено на **0.19.x**) + **Node.js**; **git** для git-фич. Любой провайдер;
  задуман для локальных моделей с маленьким контекстом. Работает на macOS, Linux и Windows.

## Установка / обновление

**Установка и обновление — это одна и та же команда**: запусти ещё раз, и оно обновится на
месте. Копирует только собственные файлы набора в `~/.qwen`; твои другие скиллы, настройки,
API-ключи и память не трогает. Кроссплатформенно: **macOS · Linux · Windows** (один Node-
установщик — сам ставит bash-бэкенды на macOS/Linux и Node-бэкенды на Windows).

(ИИ-агент? — смотри плашку в самом верху этого README / [`INSTALL_FOR_AI.md`](INSTALL_FOR_AI.md).)

Требуется: **Node.js** + **qwen-code** (и **git** для git-фич). Установщик проверяет их и
пишет, чего не хватает.

```bash
./install.sh      # macOS / Linux
install.cmd       # Windows        (или: node install.js — где угодно)
```

Затем **перезапусти qwen-code**. Чтобы обновиться позже — повтори ту же команду (или
`/toolkit-update` изнутри qwen-code). Удалить: `./uninstall.sh` / `uninstall.cmd`.

**Проверка** (после рестарта): `/skills` показывает `brainstorm, plan, implement, checkpoint,
gitflow, audit, review, commit, docs, changelog, release, toolkit-update`; `/agents manage` —
`implementer, scout, debugger, tester, researcher, verifier`; `/status` отвечает.

## Типичная сессия от начала до конца

В основном ты говоришь с ним обычным языком — скиллы и защиты срабатывают сами. Субагентов
(`implementer`, `scout`, `tester`, `researcher`, `verifier`, `debugger`) **запускает сама
модель** по ходу скилла — вручную ты их не вызываешь. Ниже — полный цикл и *когда* тянуться
за какой командой.

```text
# ── разовая настройка проекта (липкая, на проект) ──────────────────────────
> /pin model server 10.0.0.5:8080     # записать инфра-факты, которые модель забывает (в gitignore, переживает компакт)
> /maxagents 2                        # слабый/общий сервер: ограничить число параллельных субагентов
> /cover 80                           # опц.: каждая сборка — test-first, ≥80% измеренного покрытия

# ── 1. оформить задачу ─────────────────────────────────────────────────────
> /brainstorm <твоя расплывчатая идея>   # напр. «сервис-сокращалка ссылок на Python»
    ← для РАСПЛЫВЧАТОЙ идеи. Проведёт интервью, согласует объём + критерии готовности +
      краевые случаи и запишет спеку в .qwen/PROGRESS.md, чтобы пережить компакт.

> /plan добавь эндпоинт редиректа со статистикой кликов
    ← для КОНКРЕТНОЙ, но нетривиальной задачи. Исследует репо через scout-субагента и
      раскладывает на мелкие задачи по зависимостям в .qwen/PROGRESS.md. Дизайн, не код.

# ── 2. собрать ─────────────────────────────────────────────────────────────
> /dev собери                         # (или /implement — просто выполнить готовый план)
    ← режим разработки ВКЛ (на проект, липкий): модель становится АРХИТЕКТОРОМ и делегирует
      каждую задачу свежему implementer-субагенту — сама код не пишет. Тянет researcher-дайджест
      по незнакомым библиотекам, отмечает PROGRESS.md по мере готовности и закрывает независимым
      tester-субагентом, который проверяет критерии готовности буквально. Компакт посреди сборки?
      SessionStart перезагрузит PROGRESS.md; работа продолжится с первой невыполненной задачи.

> /status                             # в любой момент: какой режим, цель, следующая невыполненная задача
> /checkpoint                         # перед рискованным шагом или паузой: снапшот состояния (/checkpoint restore — вернуть)

# ── 3. проверить ───────────────────────────────────────────────────────────
> /review                             # общий ревью диффа (баги, краевые случаи) — scout'ы предлагают, verifier подтверждает
> /audit                              # ревью по безопасности (авторизация, инъекции, секреты) — каждая находка проверяется до отчёта

# ── 4. задокументировать ───────────────────────────────────────────────────
> /docs                               # синхронизировать README/доки с изменениями (в т.ч. переведённый README)
> /changelog                          # свести изменения в CHANGELOG.md (breaking-изменения первыми)

# ── 5. выкатить ────────────────────────────────────────────────────────────
> запушь готовое
    ← /commit пишет Conventional Commit; gitflow создаёт `dev` если нет и пушит в origin/dev.
      main не трогается (git-branch-guard блокирует); secret-guard не даст ключу утечь в коммит.

> выкати в main                       # main защищён — попросит сначала авторизовать
> /versioning              # (если тегаешь релизы) подтвердить схему версий
> /main-push               ← открываешь 15-минутное окно релиза
> выкати в main            ← теперь мержит dev → main и пушит
> /release                 ← нарезать тег + GitHub Release из CHANGELOG, чтобы релиз совпадал с main

> /dev off                            # обратно к одиночному агенту для быстрых вопросов
```

**Правило большого пальца:** `/brainstorm` когда идея размыта → `/plan` когда конкретна, но
крупна → `/dev` (или `/implement`) собрать → `/review` на корректность, `/audit` на
безопасность → `/docs` + `/changelog` → коммит/пуш (dev) → `/main-push` и релиз (main) → `/release` (тег+публикация). Смысл:
не нужно помнить правила веток, повторять план после компакта или следить, какой субагент что
делает — это берёт на себя тулкит.

Прочее по мере надобности: **`/bro`** меняет тон/персону (на проект, как и `/dev`);
**`/toolkit-update`** подтягивает свежий тулкит.

## Вклад в сам тулкит

Эта секция **не** про режим `/dev` — она для тех, кто правит исходники самого тулкита
(скиллы, хуки, установщик в этом репозитории) и шлёт PR. Поменял что-то здесь? Сначала
прогони тест-харнесс без зависимостей: `node test/run.js` — он проверяет allow/deny-поведение
хуков, паритет бэкендов `/pin` (bash + Node) и полный цикл установки/удаления, всё во
временных папках (твой настоящий `~/.qwen` не трогается).

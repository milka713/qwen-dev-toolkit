# qwen-dev-toolkit

Профессиональный набор рабочих практик для [qwen-code](https://github.com/QwenLM/qwen-code)
на **локальных моделях с маленьким контекстом** (например, llama.cpp-сервер с окном
~90–100k токенов). Превращает модель из «кодера, который пашет всё в одном контексте» —
и спотыкается на ~15% проекта после переполнения и лоссового компакта — в
**архитектора, который делегирует**, с durable-состоянием, переживающим компакт.

> 🇬🇧 English version: **[README.md](README.md)**.

---

## Проблема

На маленьком окне типичный провал такой:

> модель делает всю работу в основном контексте → упирается в лимит → авто-компакт →
> теряет цель, решения и то, что уже сделано → проект ломается на половине.

## Как решается — три опоры

### 1. Делегирование (чтобы контекст не забивался)
Главная сессия остаётся **архитектором**: планирует и делегирует. Вся тяжёлая работа —
чтение кучи файлов, написание кода, сборки/тесты — идёт в **одноразовых контекстах
субагентов**. Их «шум» не попадает в основной контекст, он остаётся маленьким, и проект
доходит до конца. Переполнение субагента безвредно: он возвращает то, что успел, а
архитектор переотдаёт остаток. Субагентов — **сколько нужно**, один на мелкую задачу.

### 2. Durable-состояние на диске
Прогресс живёт в `.qwen/PROGRESS.md` (цель, решения, сделано/осталось, грабли) — а не
только в переписке. Переживает рестарт, `/clear` и компакт. Хук `SessionStart`
автоматически переинжектит его в начале каждой сессии (в т.ч. сразу после компакта),
поэтому модель всегда знает, на чём остановилась.

### 3. «Прибитые» данные, неуязвимые к компакту (без переобъявления)
**Почему не нужно переобъявлять флаги после компакта?** Потому что компакт сжимает только
*историю сообщений* — он вообще не трогает файлы `QWEN.md`, которые qwen-code подгружает
как системный контекст на **каждый** запрос. Поэтому всё, что лежит в `QWEN.md`,
фактически *прибито*: всегда присутствует, никогда не сжимается, переинжект не нужен.

Именно так хранится **режим разработки**: `/dev` пишет небольшой помеченный блок в
`QWEN.md` проекта. С этого момента модель видит «режим разработки активен» на каждом ходу
и после каждого компакта, без какого-либо переобъявления. (Большой растущий `PROGRESS.md`
так *не* пинится — таскать его на каждый запрос значило бы тратить крошечный бюджет —
поэтому он использует событийный хук `SessionStart`. Маленький прибитый флаг + большое
состояние, восстанавливаемое по событию = правильное разделение.)

## Компоненты — что делает каждая часть

### Команды — детерминированные тумблеры (вызываются вручную)
Сделаны **custom-командами**, поэтому изменение файла **детерминировано** (выполняют
shell-шаг через `!{…}`, а не на усмотрение модели). Каждая пинит блок в `QWEN.md` проекта
(или `FACTS.md`), поэтому состояние переживает компакт без переобъявления.

| Команда | Что делает |
| ------- | ---------- |
| `/dev` · `on` · `off` · `status` · `<что построить>` | Переключатель режима разработки. `on` пинит блок в `QWEN.md` (сессия теперь планирует в основном контексте и делегирует субагентам); `off` убирает; `<цель>` включает **и** сразу строит. Идемпотентно; остальное содержимое `QWEN.md` сохраняется. |
| `/cover` · `on` · `off` · `status` | **Режим тестового покрытия.** Когда включён, каждая фича обязана нести реальные тесты (edge/ошибки, не только happy-path), покрытие **измеряется** реальным инструментом (`pytest --cov`, `jest --coverage`, …) и должно быть **≥90% по изменённому коду** — на выходе проверенный результат, а не «тыква», которая выглядит готовой. |
| `/pin <что угодно>` · `list` · `remove <текст>` · `clear` | Запомнить **любую** важную инфу (IP/порт сервера, команду деплоя, квирк окружения, решение, URL, сниппет) в **непробиваемый компактом `FACTS.md`** — автозагружается через `@FACTS.md` в `QWEN.md` и добавлен в **`.gitignore`** (не утечёт в репо). Всегда в контексте, никогда не сжимается. Проектный. |

### Скиллы (вызываются и моделью, и вручную)

| Скилл | Конкретно, что делает |
| ----- | --------------------- |
| `/plan` | Превращает размытый или большой запрос в **упорядоченный по зависимостям список задач** в `.qwen/PROGRESS.md`, готовый для `/implement`. Исследует код через read-only субагента `scout` (чтобы не забивать основной контекст чтениями), снимает открытые вопросы, выбирает структуру. Выдаёт план, а не код. |
| `/implement` | **Архитектор-оркестратор.** Фиксирует цель в `.qwen/PROGRESS.md`, декомпозирует работу на мелкие задачи и гоняет **каждую задачу в свежем субагенте `implementer`**, записывая результат и отмечая задачу по ходу. Завершает end-to-end проверкой **канонической** командой, как в чистом чекауте/CI. Для любой многошаговой или многофайловой сборки. |
| `/checkpoint` | **Умное durable-сжатие.** Выбирает важное (цель, решения, сделано/осталось, грабли) в `.qwen/PROGRESS.md` — отбор, а не сырой дамп — чтобы пережить лоссовое авто-сжатие движка. `/checkpoint restore` перезагружает. Запускай, когда контекст полнеет, вместо того чтобы давить до переполнения. |
| `/audit` | **Аудит безопасности на уровне архитектуры и кода.** Ищет хардкод-секреты и секреты, готовые к коммиту, затем смотрит authn/authz, инъекции (SQL/команды/пути), SSRF, небезопасную десериализацию, слабую крипту, CORS/конфиг и рискованные зависимости — выдаёт находки по серьёзности и чинит безопасные. Запускай перед выкаткой security-чувствительной работы. |

### Субагенты (делегированные исполнители с изолированным контекстом)

| Субагент | Конкретно, что делает |
| -------- | --------------------- |
| `implementer` | Берёт **одну** задачу и доводит до рабочего, **проверенного** состояния в своём контексте: читает реальные файлы, держит конвенции, реализует полностью (без заглушек) и проверяет **канонической** командой из корня репозитория — чиня упаковку, если проверка проходит только через env-трюк. Возвращает краткое резюме; тяжёлый «шум» не попадает в основной контекст. |
| `scout` | **Read-only** исследователь кода. Отвечает на точечный вопрос «где живёт X / что затронет изменение» и возвращает **компактный дайджест** (ключевые файлы, как связаны, конвенции, реальные команды сборки/тестов), чтобы основная сессия не читала файлы пачками. |

### Хуки (`~/.qwen/settings.json`)

| Хук | Конкретно, что делает |
| --- | --------------------- |
| `SessionStart` → `session-start-restore.js` | Если есть `.qwen/PROGRESS.md`, инжектит его в начале сессии (и после компакта/resume), чтобы модель автоматически восстановила цель/решения/следующие шаги. |
| `PreCompact` → `pre-compact-steer.js` | Срабатывает прямо перед встроенным сжатием и направляет его **сохранять** цель, решения, список файлов и разбивку сделано/осталось, а не выкидывать их. |
| `PreToolUse` → `secret-guard.js` | **Детерминированно блокирует** любой `write_file`/`edit`/`run_shell_command`, в содержимом которого есть хардкод-секрет (приватные ключи, токены AWS/OpenAI/GitHub/Slack/HF/…, высокоэнтропийные присваивания), или который стейджит/коммитит явный секрет-файл (`.env`, `id_rsa`, `*.pem`). Принуждает «не держать ключи в коде, не выкладывать в репо» на уровне движка, а не на доверии к модели. Мало ложных срабатываний: использование env-переменных и плейсхолдеры проходят. |

### Контекст-гайд (`QWEN.md`)
Лаконичное глобальное правило работы в `~/.qwen/QWEN.md`: два режима (обычный и
разработки) и **дисциплина памяти** — запоминать долговечные факты (IP/порты серверов,
команды деплоя/запуска, квирки окружения, где лежат креды, устойчивые предпочтения),
а *не* временное состояние задач и не то, что выводится из репозитория. Нативная
авто-память включена.

---

## Два режима

- **Обычный режим (по умолчанию):** модель отвечает одним агентом. Подходит для вопросов,
  объяснений, мелких правок, изменений в один-два файла — без субагентов и накладных
  расходов.
- **Режим разработки:** включается один раз на проект через `/dev` (или `/dev <что
  построить>`). Тогда модель планирует в основном контексте и делегирует **всю** реализацию
  субагентам `implementer`. Флаг прибит в `QWEN.md`, поэтому остаётся включённым всю сессию
  **и после любого компакта или рестарта**, пока не выполнишь `/dev off`.

## Где что лежит (области видимости)

`QWEN.md` — это **контекст-файл**, который qwen-code заново подключает как системный
контекст на каждый запрос (поэтому он не сжимается). Грузится из двух мест —
**сессионного `QWEN.md` не существует**:

- `~/.qwen/QWEN.md` → **глобальный** (любой проект)
- `<проект>/QWEN.md` и вверх по дереву → **проектный** (папка, где запущен qwen)

Тулкит кладёт каждый вид состояния туда, где ему место:

| Состояние | Где лежит | Область |
| --------- | --------- | ------- |
| Скиллы, субагенты, команды, хуки, базовый гайд | `~/.qwen/…` | **Глобально** — во всех проектах |
| Память `/pin` (IP/порт, решения, что угодно) | `<проект>/FACTS.md` (+ `@FACTS.md` в проектном `QWEN.md`), в `.gitignore` | **Проектно** |
| Флаги `/dev` и `/cover` | блок в `<проект>/QWEN.md` | **Проектно**, липко до `/dev off` / `/cover off` |
| Durable-состояние задач | `<проект>/.qwen/PROGRESS.md` | **Проектно** |

То есть сам тулкит — глобальный, твоя память и флаги — на проект, а всё, что должно
пережить компакт, лежит в файле, загружаемом как `QWEN.md`, а не в переписке.

## Установка

**Одной командой (рекомендуется) — полная установка в `~/.qwen`:**

```bash
./install.sh
```

Копирует скиллы, субагентов, команду `/dev` и хук-скрипты в `~/.qwen`, затем идемпотентно
мёржит настройки хуков и авто-памяти в `~/.qwen/settings.json` и гайд в `~/.qwen/QWEN.md`.
Не трогает твои ключи, env и прочие настройки; безопасно запускать повторно. После —
перезапусти qwen-code.

**Как нативный extension qwen-code** (скиллы + субагенты + команда `/dev` + гайд):

```bash
qwen extensions link /путь/к/qwen-dev-toolkit          # локальный чекаут
# или из git-репозитория:
qwen extensions install https://github.com/milka713/qwen-dev-toolkit
```

Авто-хуки компакта лежат в `settings.json` и механизмом extension'а не переносятся —
дополнительно запусти `./install.sh` (или скопируй `hooks/*.js` в `~/.qwen/hooks/` и добавь
две записи хуков в `~/.qwen/settings.json`; точный вид — в `install.sh`), чтобы их включить.

**Вручную:** скопируй `skills/*` → `~/.qwen/skills/`, `agents/*` → `~/.qwen/agents/`,
`commands/*` → `~/.qwen/commands/`, `hooks/*` → `~/.qwen/hooks/`, и слей `QWEN.md`.

### Проверка

Перезапусти qwen-code, затем:

- `/skills` → `implement`, `checkpoint`, `plan`
- `/agents manage` → `implementer`, `scout`
- `/dev status` → команда `/dev` отвечает
- `/dev`, потом `/plan собери простой CLI todo` → в `.qwen/PROGRESS.md` появляется план

## Использование

```text
/dev                                  # вход в режим разработки (прибит в QWEN.md)
/cover on                             # требовать реальные тесты, ≥90% покрытия
/pin модель-сервер 10.0.0.5 порт 8080 # пин инфра-фактов (компакт-пруф, в .gitignore)
/plan добавь JWT-авторизацию в API    # дизайн → .qwen/PROGRESS.md
/implement                            # выполнить план через субагентов
/audit                                # аудит безопасности перед выкаткой
/checkpoint                           # сохранить состояние, когда контекст полнеет
/checkpoint restore                   # перезагрузить состояние после рестарта/компакта
/dev off                              # обратно в обычный режим одного агента
```

Или одной строкой: `/dev собери Python CLI трекер расходов с SQLite и pytest` — включает
режим разработки и запускает всю делегированную сборку.

## Требования

- qwen-code (проверено на **0.19.x**) и Node.js (qwen-code и так его требует).
- Работает с любым провайдером; задуман для локальных моделей с маленьким контекстом, где
  это критичнее всего.

## Удаление

```bash
./uninstall.sh
```

Удаляет только файлы и блоки конфига этого набора. Остальные настройки, env, память и файлы
`.qwen/PROGRESS.md` остаются нетронутыми.
